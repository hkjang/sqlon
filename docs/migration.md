# JAMYPG 데이터의 SQLON 마이그레이션

SQLON을 데이터 경로 옵션 없이 처음 시작하면 같은 상위 디렉터리의 기존
`data/metadb`를 탐색합니다. 이전 데이터가 있고 `data/sqlon`에 없는 파일이
있을 때만 백업을 먼저 만든 뒤 SQLON 디렉터리로 병합합니다.

```sh
./sqlon -transport http -addr 127.0.0.1:9797
```

## 보존 및 충돌 규칙

- `db_profiles.json`, 카탈로그 데이터셋, 감사 JSONL을 포함해 기존 디렉터리의
  일반 파일을 상대 경로 그대로 보존합니다.
- `data/sqlon`에 이미 같은 경로의 파일이 있으면 SQLON 파일이 우선하며 기존
  파일로 덮어쓰지 않습니다.
- 변경 전에 `data/backups/sqlon-pre-migration-<UTC 시각>/legacy`에 기존
  JAMYPG 데이터를 모두 백업합니다. 기존 SQLON 디렉터리도 있었다면 같은 백업의
  `sqlon`에 함께 보존합니다.
- 준비가 끝난 디렉터리를 원자적으로 교체하며, 실패하면 기존 SQLON 디렉터리를
  다시 활성화합니다.
- `.sqlon-migration.json`에 마이그레이션 시각, 원본 경로와 복사 파일 수를
  기록합니다. 이후 실행은 빠진 파일이 없으면 아무것도 변경하지 않습니다.
- 심볼릭 링크와 일반 파일이 아닌 항목은 안전을 위해 거부합니다.

## 자동 마이그레이션을 사용하지 않는 경우

`-data`, `SQLON_DATA_DIR` 또는 한 릴리스 동안 지원하는
`JAMYPG_DATA_DIR`를 지정하면 해당 경로를 그대로 사용하며 자동 마이그레이션을
수행하지 않습니다.

```sh
SQLON_DATA_DIR=/srv/sqlon ./sqlon
./sqlon -data /srv/sqlon
```

이는 외부 볼륨이나 별도 변경 절차가 있는 운영 환경에서 예기치 않은 데이터
이동을 방지하기 위한 동작입니다.

## 확인과 복구

시작 로그의 `SQLON data migrated` 메시지에서 대상과 백업 경로를 확인합니다.
기동 후 프로파일, 카탈로그 상태, 과거 감사 로그를 확인한 다음 백업 보존 정책을
적용합니다.

복구가 필요하면 SQLON을 중지하고 현재 `data/sqlon`을 별도 보존한 뒤, 로그에
표시된 백업의 `sqlon`(기존 SQLON이 있던 경우) 또는 `legacy` 내용을 운영자가
검증한 빈 대상 디렉터리에 복사합니다. 자동 마이그레이션은 백업을 삭제하지
않습니다.
