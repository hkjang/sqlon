## 1. Executive Summary (요약 보고)

본 보고서는 금융데이터의 안전한 탐색과 생성형 AI 기반 대화형 질의 채널 구축을 위해 AI 인프라실에서 연구·개발한 **JAMYPG NL2SQL MCP(Model Context Protocol)** 시스템의 도입 성과 및 전사 확산 로드맵을 기술합니다. JAMYPG는 기존 Oracle 전용 jasql 프로젝트를 PostgreSQL·MySQL·MariaDB 3종 오픈소스 DBMS 대상으로 전환·확장한 후속 시스템입니다.

JAMYPG는 기존 Text2SQL 기술이 가진 보안 한계와 할루시네이션(환각) 리스크를 원천 차단하기 위해 <strong>"LLM에게 데이터베이스 스키마 전체를 전달하지 않는다"</strong>는 원칙 하에 설계되었습니다. 메타데이터 카탈로그 컴파일, 다층 정적 가드레일, 실측 실행계획 분석을 통합 가동하여 금융권 컴플라이언스를 만족하는 무결한 데이터 조회 체계를 완성하였습니다. 

최근 버전(v0.12.0+)에 탑재된 <strong>GraphRAG 기반 컨텍스트 검색 계층(`retrieve_context`)</strong>을 활용한 골든셋 80개 케이스 실측 결과, <strong>테이블 검색 Recall@5 정확도 95.0%, 컬럼 재현율 91.0%, 조인 경로 정합성 72.0%, 값 증거 매칭률 84.0%</strong>를 확보하며 업계 최상위권 데이터 그라운딩 성과를 확인하였습니다. 이를 통해 데이터 추출 리드타임을 기존 평균 48시간에서 **5분 이내**로 단축시켜 전사 데이터 활용 생산성을 혁신할 준비를 마쳤습니다.

## 2. 도입 배경 및 비즈니스 목적

### 2.1. 기존 생성형 AI Text2SQL 기술의 한계
생성형 AI(LLM)의 출현으로 자연어 질문을 SQL 쿼리로 자동 변환하는 시도가 활발하나, 금융 도메인의 복잡성과 정보 보호 규제 하에서는 다음과 같은 세 가지 심각한 리스크가 존재합니다.

1. **스키마 과부하 및 높은 토큰 비용**:
   데이터베이스의 수백 개 테이블 구조를 프롬프트에 직접 주입할 시 토큰 비용이 기하급수적으로 증가하며, 관련 없는 스키마 정보가 섞여 모델의 집중력이 저하되고 쿼리 품질이 크게 하락합니다.
2. **금융 데이터 보안(PII) 유출 위험**:
   고객명, 계좌번호, 주민등록번호 등 개인식별정보(PII) 컬럼이 정제 없이 LLM으로 유입되거나 쿼리 결과에 직접 노출되는 등 개인정보 보호법 위반 가능성이 항상 상존합니다.
3. **할루시네이션에 의한 쿼리 오작동 및 시스템 부하**:
   모델이 존재하지 않는 테이블/컬럼명을 조작하여 질의하거나, 비효율적인 다중 조인(Join) 또는 Full Table Scan을 유발하는 폭주성 쿼리를 가동하여 데이터베이스 서버 장해를 유발할 수 있습니다.

### 2.2. JAMYPG 도입의 목적
- **거버넌스 확립**: 카탈로그 기반의 정적 검증 계층을 두어 비인가/위험 쿼리를 실행 전에 차단합니다.
- **의사결정 속도 단축**: 현업 담당자가 데이터 추출 요건을 분석가에게 의뢰하고 대기하는 병목 구간을 제거하여 실시간 데이터 드리븐 의사결정을 실현합니다.
- **인프라 비용 효율화**: 별도의 고비용 벡터 데이터베이스(Vector DB) 서버 구축 없이, 경량화된 인메모리 컴파일 기술을 통해 신속하고 가볍게 구동 가능한 고성능 NL2SQL 허브를 확보합니다. 아울러 순수 Go 단일 바이너리 구조로 상용 DB 클라이언트 라이브러리 의존을 제거하여 배포 비용을 절감합니다.

## 3. JAMYPG 핵심 기술 강점 (5대 기술적 축)

JAMYPG는 AI 인프라실이 개발한 5대 핵심 축을 바탕으로 정밀하고 안전한 그라운딩(Grounding) 환경을 제공합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            JAMYPG 5대 핵심 아키텍처                          │
├─────────────────┬─────────────────┬─────────────────┬───────────┬───────────┤
│    그라운딩     │    사전 우선    │  압축 컨텍스트  │ 다층 검증 │ 환류 학습 │
│  (Catalog-only) │ (Dict-priority) │   (Compress)    │  (Guard)  │  (Loop)   │
└─────────────────┴─────────────────┴─────────────────┴───────────┴───────────┘
```

1. **카탈로그 및 GraphRAG 그라운딩 (Grounding)**
   - LLM은 내부 카탈로그 엔진에 컴파일되어 존재하는 개체명만을 사용하며, 검색 시 **시드 검색 → 1-hop 조인그래프 확장 → 데이터 값 증거 및 usage prior 하이브리드 재정렬** 과정을 거치는 GraphRAG 기술을 연계 구동하여 스키마 맥락을 완벽히 복원합니다.
2. **비즈니스 지표 사전 우선 적용 (Dictionary First)**
   - 연체율, 평균 신용점수 등 금융 통계의 분모·분자 구성이 복잡한 지표는 LLM이 임의로 식을 생성하지 못하도록 차단합니다. 지표 사전(`metrics.json`)에 등록된 공인 수식(`Expression`)을 강제 바인딩하여 수치의 일관성을 확보합니다.
3. **컨텍스트 압축 (Compressed Context)**
   - 질문과 연관된 테이블과 컬럼만을 다중 스코어링으로 추려내어, 수십 바이트 수준으로 압축된 컴팩트 스키마 정보만을 LLM에 제공함으로써 할루시네이션 발생 여지를 원천 제어합니다.
4. **다층 보안 가드레일 (Multi-layered Guardrails)**
   - SELECT/WITH 문 외의 DML/DDL 유입 차단, PII 컬럼 탐지 및 마스킹, 카티션 조인 위험 제어 등 33종의 정적 룰 검증을 수행하고, 실패 시 에러 코드와 자가 수정용 피드백 힌트(`fix_hints`)를 제공합니다. 실행 계층에서는 DSN 수준 read-only 세션 강제(PostgreSQL `default_transaction_read_only=on` 등)가 이중으로 가동됩니다.
5. **환류 자동 학습 루프 (Feedback Learning Loop)**
   - 사용자가 되묻기(Clarification)에 답변하여 성공적으로 해소되면 해당 결과를 즉시 학습 피드백으로 원자적 적재하여 다음 카탈로그 리로드 시 검색 Usage Prior 가중치로 환류시킵니다.
   - 반복적으로 느린 쿼리를 유발하는 테이블에 검색 감점을 적용하고, 오류가 잦은 컬럼의 대체재 힌트를 learned 룰로 자동 승격 적용합니다.

## 4. 정량적/정성적 도입 성과

### 4.1. 정확도 지표 평가 (80개 기준 케이스 검증 결과)

AI 인프라실은 대표적인 데이터셋 질문-SQL 80개 세트를 구축하여 정확도 회귀 평가를 가동했습니다. 실측된 정확도 성과는 사전에 설정한 파이프라인 통과 임계치를 대폭 상회합니다.

| 평가 지표명 | 지표 정의 | 목표 기준 | **실측 성과** | 달성도 |
| :--- | :--- | :---: | :---: | :---: |
| **테이블 검색 정확도 (Recall@5)** | 기대 테이블이 GraphRAG 검색 탑 5위 풀 내에 포함된 비율 | ≥ 85.0% | **95.0%** | **초과 달성** |
| **컬럼 매칭 재현율** | 필요한 컬럼들이 매칭 리스트 상에 노출된 평균 비율 | ≥ 80.0% | **91.0%** | **초과 달성** |

### 4.2. 인프라 운영성 및 자원 효율성
- **초고속 응답성**: 모든 메타데이터를 기동 시 메모리에 적재 및 인덱싱하므로, 도구 호출 연산 소요 시간이 **평균 수십 밀리초(ms) 이내**로 완료되어 대화형 실시간 AI 에이전트에 최적입니다.
- **인프라 비용 혁신**: 실시간 검색을 위한 값비싼 외산 벡터 DB 및 그래픽 연산 장치(GPU) 없이 가볍게 구동 가능한 자립형 아키텍처로 인프라 유지 비용을 획기적으로 낮췄으며, CGO·상용 클라이언트 라이브러리 없는 순수 Go 단일 이미지 배포로 폐쇄망 반입 절차도 대폭 간소화하였습니다.

## 5. 거버넌스 및 금융 컴플라이언스 관리 체계

JAMYPG는 전자금융거래법 및 개인정보보호법 등 엄격한 금융 보안 가이드를 완벽하게 수행할 수 있도록 감시 통제 장치를 포함하고 있습니다.

- **읽기 전용 정책의 물리적 강제 (3중 방어)**:
  SELECT 및 WITH로 기동되는 질의만 통과시키며, INSERT, UPDATE, DELETE, ALTER 등 데이터 오염 위험이 있는 키워드 발견 시 즉각 기각 처리합니다. 여기에 접속 세션 자체를 read-only로 강제(PostgreSQL `default_transaction_read_only=on`, MySQL `transaction_read_only=1`, MariaDB `tx_read_only=1`)하고, SELECT 전용 DB 계정 사용을 권고하여 3중 방어선을 구축합니다.
- **PII 민감 정보의 안전한 제어**:
  주민등록번호, 고객명, 계좌번호 등 마스킹 대상 속성이 SELECT에 탐지되면 검증기(`validate_sql`)가 즉각 `PII_COLUMN` 에러를 반환하여 쿼리 발행을 가로막고, 집계 형태(Count 등)로 변환하거나 마스킹을 부과하도록 힌트를 출력합니다.
- **실측 실행계획 분석 게이트 (JSON EXPLAIN 연동)**:
  접속 계정의 정책과 별개로, 쿼리가 유발할 데이터베이스 시스템 부하를 실행 전에 실측합니다. `explain_sql`이 PostgreSQL은 `EXPLAIN (FORMAT JSON)`, MySQL/MariaDB는 `EXPLAIN FORMAT=JSON`을 가동하여 Full Scan, Cartesian Product 의심 조인, 대형 정렬, Cost 지수 폭증 등 고위험 요인 탐지 시 실행을 원천 차단합니다.
- **실시간 감사 이력 적재**:
  모든 MCP 도구 호출 이력과 SQL 해시, 소요 시간, 예외 코드 등을 `audit/` 내에 일자별 감사 파일(`query-YYYYMMDD.jsonl`)로 영속 적재하여 완벽한 사후 추적성을 보장합니다.

## 6. 비즈니스 파급 효과 (ROI)

1. **데이터 분석 리드타임 혁신**:
   기존 업무 부서의 요건 정의 ➜ IT 부서 접수 ➜ 개발자 작성 ➜ 검증 및 추출 프로세스로 최소 48시간 이상 대기하던 리드타임을 자연어 질의를 통해 **5분 이내 즉시 추출**로 변환하여 실시간 비즈니스 의사결정이 가능합니다.
2. **IT 데이터 전담 인력 생산성 향상**:
   분석가들이 단순 반복성 데이터 추출 쿼리 수작업 요청에서 해방되어, AI 모델링 및 가치 중심의 비즈니스 분석 업무에 집중할 수 있는 환경을 유도합니다 (생산성 약 40% 이상 향상 기대).
3. **거버넌스 위반 리스크 Zero화**:
   수작업 추출 과정에서 발생할 수 있는 개인정보의 부주의한 노출, 비가드로 인한 대형 쿼리 폭주 사고 위험을 사전에 격리 통제하여 보안 규제 위반에 따른 손실 리스크를 차단합니다.
